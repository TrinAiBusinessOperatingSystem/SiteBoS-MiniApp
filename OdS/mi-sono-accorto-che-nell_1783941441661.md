# 📐 ORDINE DI SERVIZIO: MI-SONO-ACCORTO-CHE-NELL

## 📝 CONTESTO OPERATIVO
Attualmente, l'Assistente Sicurezza carica e visualizza indistintamente tutte le procedure operative (SOP) e i servizi registrati nel sistema, sovraccaricando l'interfaccia e l'assistente con informazioni non pertinenti alla salute e sicurezza. L'obiettivo è introdurre un meccanismo di selezione granulare, gestibile direttamente dall'amministratore durante la creazione o la modifica di un prodotto/servizio nel catalogo, per decidere esplicitamente quali elementi far confluire nell'Assistente Sicurezza.

## 🔗 MAPPATURA CONNESSIONI
Il flag booleano viene configurato graficamente su 'edit-product.html' e inviato al backend; il workflow 'saveProduct_processor.json' lo persiste nella collezione 'service_catalog' dentro 'operations.show_in_security_assistant'. Quando l'utente apre l'interfaccia dell'assistente ('assistente-sicurezza.html'), viene invocato il workflow 'Secure_Assistant.json' che esegue una query filtrata su 'service_catalog' e, a cascata, estrae ed interroga solo i blueprint correlati da 'process_blueprints' tramite corrispondenza di SKU.

## 📐 DECISIONI ARCHITETTURALI
Si è scelto di posizionare il flag all'interno della struttura esistente 'operations' nel catalogo dei servizi per preservare la coerenza semantica del database MongoDB. Il filtraggio dei dati viene delegato interamente al backend (query a livello di database in n8n) anziché al frontend: questo ottimizza le prestazioni riducendo drasticamente il payload di rete, minimizza il consumo di token nel contesto dell'assistente basato su LLM e garantisce la robustezza logica del sistema visualizzando solo dati pertinenti.

## 📐 1. SPECIFICHE FRONTEND
Nel file 'telegram_control/edit-product.html', all'interno della sezione dedicata alle opzioni operative o impostazioni (vicino ai controlli esistenti come 'Richiede Prenotazione'), aggiungere un interruttore grafico (toggle switch o checkbox) con etichetta 'Mostra in Assistente Sicurezza' associato all'ID 'show_in_security_assistant'. Aggiornare la logica JavaScript di caricamento dati affinché recuperi la proprietà booleana 'operations.show_in_security_assistant' e imposti di conseguenza lo stato del controllo visivo (se non definita, considerarla predefinita a false). Nella logica di salvataggio (invio form), includere lo stato booleano del controllo nel payload inviato al backend sotto la chiave 'operations.show_in_security_assistant'. Nel file 'telegram_control/catalog.html', nella tabella o griglia dei servizi, aggiungere un elemento visivo (come un'icona a forma di scudo o un badge colorato) che indichi a colpo d'occhio se il servizio è abilitato per l'Assistente Sicurezza. Nel file 'telegram_control/agents/assistente-sicurezza.html', assicurarsi che la visualizzazione dei servizi e delle relative procedure operative standard (SOP) si adegui alla lista già filtrata restituita dal backend, evitando di mostrare elementi esclusi dal catalogo.

## ⚙️ 2. GUIDA PASSO-PASSO BACKEND (n8n)
Nel workflow 'SiteBoS-App-Hook/catalog/editProduct.json', garantire che la query di lettura su MongoDB per la collezione 'service_catalog' includa ed esporti correttamente il campo 'operations.show_in_security_assistant' verso il client. Nel workflow 'SiteBoS-App-Hook/catalog/saveProduct_processor.json', inserire un nodo di validazione/assegnazione per catturare il valore del campo 'operations.show_in_security_assistant' proveniente dal body della richiesta; utilizzare un'espressione per normalizzare il valore a booleano puro (gestendo i casi in cui viene trasmesso come stringa 'true'/'on' o omesso in caso di checkbox non selezionata). Configurare il nodo di aggiornamento MongoDB per eseguire un operatore '$set' specifico su 'operations.show_in_security_assistant'. Nel workflow 'SiteBoS-App-Hook/intelligence/Secure_Assistant.json', modificare il nodo di query MongoDB della collezione 'service_catalog' impostando il filtro '{ "operations.show_in_security_assistant": true }'. Successivamente, inserire un nodo di elaborazione (es. Code o Item Lists) per estrarre tutti i codici SKU ('identity.item_sku') dei servizi così filtrati. Infine, modificare la query della collezione 'process_blueprints' (SOP) affinché recuperi solo i blueprint associati ai servizi selezionati tramite l'operatore '{ "service_sku": { "$in": <array_di_sku_filtrati> } }'.

## 💾 3. STRUTTURA DATABASE (MongoDB)
Modifica alla collezione 'service_catalog' (Database: MemoryManager): inserimento del nuovo campo booleano 'operations.show_in_security_assistant' con tipo 'Boolean' (valore predefinito consigliato: false o null). Nelle query di scrittura/aggiornamento del catalogo, utilizzare l'operatore '$set' per aggiornare selettivamente 'operations.show_in_security_assistant' senza intaccare altri campi dell'oggetto 'operations'. Interrogazione su 'process_blueprints' (Database: MemoryManager) tramite filtro relazionale dinamico basato sul campo 'service_sku' incrociato con i risultati attivi di 'service_catalog'.
